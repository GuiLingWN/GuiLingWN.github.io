# MyBatis使用技巧与常见问题

MyBatis作为一款优秀的持久层框架，在使用过程中有一些需要注意的地方。本文总结了MyBatis开发中的常见问题和解决方案。

# ${}和#{}的区别

在MyBatis的SQL映射文件中，参数有两种引用方式：#{}和${}，它们的区别非常重要。

# 主要区别

  1. 预编译机制

    • #{}：使用预编译方式，会被替换成?占位符，然后通过PreparedStatement设置参数
    • ${}：是字符串替换，直接将参数值拼接到SQL中

  2. SQL注入风险

    • #{}：由于使用预编译，可以有效防止SQL注入攻击
    • ${}：直接拼接字符串，存在SQL注入风险

  3. 使用场景

    • #{}：用于传递参数值，适用于大多数场景
    • ${}：用于动态SQL片段，如表名、字段名等无法使用占位符的场景

# 使用建议

  • 优先使用#{}方式传递参数
  • 只有在必须动态拼接SQL结构时才使用${}
  • 使用${}时要特别注意参数来源，避免SQL注入风险

# @Param注解使用问题

# 问题描述

在MyBatis接口方法中传递参数时，如果不使用@Param注解，可能会遇到以下错误：

org.apache.ibatis.binding.BindingException: Parameter '参数名' not found.

# 解决方案

有以下几种方式可以解决这个问题：

# 方式一：使用@Param注解（推荐）

// Mapper接口
List<User> selectByNameAndAge(@Param("name") String name, @Param("age") Integer age);

<!-- XML映射文件 -->
<select id="selectByNameAndAge" resultType="User">
    SELECT * FROM user WHERE name = #{name} AND age = #{age}
</select>

# 方式二：使用单个对象参数

如果参数较多，可以封装成一个对象：

List<User> selectByCondition(UserQuery query);

<select id="selectByCondition" resultType="User">
    SELECT * FROM user WHERE name = #{name} AND age = #{age}
</select>

# 方式三：使用Map传参

List<User> selectByMap(Map<String, Object> params);

# 最佳实践

  1. 单个参数：可以不使用@Param，直接在SQL中引用
  2. 多个参数：建议使用@Param注解明确指定参数名
  3. 参数较多：封装成对象或使用Map传递

使用@Param注解的好处是代码更加清晰，不容易出错，特别是在多人协作的项目中。

# MyBatis Generator代码生成

MyBatis Generator是一个可以自动生成MyBatis映射文件和实体类的工具，可以大大提高开发效率。

# 使用步骤

  1. 配置generatorConfig.xml文件
  2. 配置数据库连接信息
  3. 配置要生成的表信息
  4. 运行Generator生成代码

详细配置和使用方法可以参考：MyBatis Generator详细教程 (opens new window)

# 小结

MyBatis的使用需要注意参数传递方式的选择和SQL注入防护。合理使用#{}和${}，正确添加@Param注解，可以避免很多常见问题。同时，利用MyBatis Generator等工具可以提高开发效率，减少重复劳动。
