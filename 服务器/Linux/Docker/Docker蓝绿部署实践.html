# Docker蓝绿部署实践

在生产环境中，我们经常需要在不中断服务的情况下更新应用。蓝绿部署是一种常见的零停机部署方案。本文分享一个基于Docker的蓝绿部署脚本实现。

# 什么是蓝绿部署

蓝绿部署是一种应用发布策略，通过维护两套完全相同的生产环境（蓝环境和绿环境），在发布新版本时：

  1. 在空闲环境中部署新版本
  2. 测试新版本正常运行
  3. 将流量切换到新版本
  4. 停止旧版本容器

这样可以实现无缝切换，出现问题时也能快速回滚。

# 实现思路

在Docker环境下，我们可以通过端口映射来实现简化版的蓝绿部署：

  1. 检测当前运行中的容器及其使用的端口
  2. 构建新镜像
  3. 选择一个空闲端口启动新容器
  4. 验证新容器启动成功
  5. 停止并删除旧容器

# 脚本实现

#!/bin/bash

# 设置容器和镜像的基础名称
BASE_CONTAINER_NAME="ar_guide"
IMAGE_NAME="ar_guide_image"
CONTAINER_PORT="12584"
NEW_EXTERNAL_PORT=0

# 查找具有相同基础名称的正在运行的容器，并获取其端口映射
RUNNING_CONTAINER_ID=$(docker ps -q -f name="$BASE_CONTAINER_NAME")

if [ -n "$RUNNING_CONTAINER_ID" ]; then
    RUNNING_CONTAINER_NAME=$(docker inspect --format '{{ .Name }}' $RUNNING_CONTAINER_ID | sed 's/\\///')
    echo "找到正在运行的容器，名称为: $RUNNING_CONTAINER_NAME"
    CUR_EXTERNAL_PORT=$(docker port $RUNNING_CONTAINER_ID $CONTAINER_PORT/tcp | cut -d: -f2)
    echo "现有容器使用的外部端口: $CUR_EXTERNAL_PORT"
else
    echo "没有找到基础名称为: $BASE_CONTAINER_NAME 的正在运行的容器"
fi

# 构建新的镜像
echo "正在构建新镜像..."
docker build -t $IMAGE_NAME .

# 查找一个可用端口
for port in {12581..12584}; do
    if ! ss -tuln | grep -q ":$port"; then
        NEW_EXTERNAL_PORT=$port
        break
    fi
done

if [ $NEW_EXTERNAL_PORT -eq 0 ]; then
    echo "在范围内没有找到可用端口。"
    exit 1
fi

# 使用外部端口生成新的容器名称
NEW_CONTAINER_NAME="${BASE_CONTAINER_NAME}_${NEW_EXTERNAL_PORT}"

# 使用可用端口运行新容器
echo "使用名称: $NEW_CONTAINER_NAME 和端口: $NEW_EXTERNAL_PORT 运行新容器"
docker run --restart=always -d \
  -p $NEW_EXTERNAL_PORT:$CONTAINER_PORT \
  --name $NEW_CONTAINER_NAME \
  -v /software/logs/"$NEW_CONTAINER_NAME":/logs/"$NEW_CONTAINER_NAME" \
  --log-opt max-size=1m \
  --log-opt max-file=5 \
  -e TZ="Asia/Shanghai" \
  $IMAGE_NAME

# 等待新容器启动
echo "等待新容器启动..."

# 检查新容器是否正在运行
NEW_CONTAINER_ID=$(docker ps -q -f name=$NEW_CONTAINER_NAME)
if [ -n "$NEW_CONTAINER_ID" ]; then
    echo "新容器成功启动，ID: $NEW_CONTAINER_ID"

    # 如果旧容器存在，则停止并移除它
    if [ -n "$RUNNING_CONTAINER_ID" ]; then
        echo "停止并移除旧容器，ID: $RUNNING_CONTAINER_ID"
        docker rm -f $RUNNING_CONTAINER_ID
    fi
else
    echo "新容器启动失败。退出..."
    exit 1
fi

echo "部署完成。"

# 脚本功能说明

# 核心功能

  1. 端口管理

    • 自动检测当前容器使用的端口
    • 从预设端口范围(12581-12584)中查找可用端口
    • 使用端口号作为容器名称的一部分，便于管理

  2. 镜像构建

    • 每次部署前重新构建镜像
    • 确保使用最新代码

  3. 容器配置

    • 自动重启策略：--restart=always
    • 日志限制：防止日志文件无限增长
    • 时区设置：使用Asia/Shanghai时区
    • 日志挂载：便于查看和管理日志

  4. 安全切换

    • 只有在新容器成功启动后才删除旧容器
    • 如果新容器启动失败，保留旧容器继续运行
    • 避免服务中断

# 使用方式

  1. 将脚本保存为deploy.sh
  2. 在Dockerfile所在目录执行：
    chmod +x deploy.sh
./deploy.sh

# 适用场景

这个脚本适合：

  • 单机Docker部署
  • 后端API服务
  • 内部工具服务
  • 开发/测试环境

不适合：

  • 需要负载均衡的高并发场景（建议使用Kubernetes）
  • 需要数据库迁移的复杂部署
  • 有状态服务（需要额外考虑数据持久化）

# 改进建议

# 添加健康检查

在删除旧容器前，应该验证新容器的健康状态：

# 等待并检查容器健康状态
for i in {1..30}; do
    if docker exec $NEW_CONTAINER_NAME curl -f http://localhost:$CONTAINER_PORT/health; then
        echo "新容器健康检查通过"
        break
    fi
    if [ $i -eq 30 ]; then
        echo "新容器健康检查失败"
        docker rm -f $NEW_CONTAINER_ID
        exit 1
    fi
    sleep 2
done

# 添加回滚机制

保存旧容器ID，在新容器失败时自动回滚：

# 发现新容器异常时
if [ -n "$RUNNING_CONTAINER_ID" ]; then
    echo "新容器异常，回滚到旧版本"
    docker start $RUNNING_CONTAINER_ID
fi

# 集成到CI/CD

将脚本集成到GitLab CI或Jenkins中：

# .gitlab-ci.yml示例
deploy:
  stage: deploy
  script:
    - scp -r ./* server:/app/
    - ssh server "cd /app && ./deploy.sh"
  only:
    - master

# 注意事项

  1. 端口范围： 根据实际情况调整可用端口范围
  2. 日志路径： 确保日志目录有足够的磁盘空间
  3. 网络配置： 如果使用Nginx反向代理，需要配置upstream动态更新
  4. 资源限制： 生产环境建议添加CPU和内存限制
  5. 备份策略： 镜像构建前建议备份重要数据
