# 一次挖矿病毒的排查经历与思考

服务器安全是每个运维人员的必修课。最近，我处理了一起典型的挖矿病毒入侵事件。服务器表现为 CPU 占用率持续 100%，通过一系列排查，最终定位并清除了病毒。我将整个过程记录下来，希望能为遇到类似问题的朋友提供一些排查思路和安全加固的建议。

# 现象：CPU 占用率飙升

一切始于监控系统发出的告警：一台服务器的 CPU 使用率长时间维持在 100%。登录服务器后，执行 top 命令，果然发现一个或多个可疑进程占用了几乎所有的 CPU 资源。

# 排查步骤

面对这种情况，我们的核心目标是：找到并理解这个异常进程，然后清除它以及它的持久化机制。

# 1. 定位异常进程

top 命令是我们的第一利器。

top

进入 top 界面后，可以按 P（大写）让进程按 CPU 使用率降序排列，这样占用最高的进程就会显示在最上方。但默认情况下，top 显示的命令名可能会被截断，我们需要看到完整的命令行才能获取更多信息。

  • 在 top 界面，按下 c 键，可以切换显示完整的命令行。

通过完整的命令行，我发现了一个名为 networkSync 的可疑进程。它的命令行参数非常扎眼：

/tmp/networkSync -o stratum+tcp://auto.c3pool.org:19999 -u 457KQKqFtyg9WtcHQdvok4N6uRZstLmPzFrPv2ewNjHeSRP1MyFFw2zHHjCpm7JoYgMGSpwCw+
  • /tmp/networkSync：病毒主体放在了 /tmp 目录下，这是一个常见的藏匿手法，因为 /tmp 目录通常权限宽松。
  • stratum+tcp://auto.c3pool.org:19999：这明显是一个矿池的地址。
  • -u 457K...：后面跟着的长字符串，毫无疑问是挖矿用的钱包地址。

至此，基本可以断定这是一起挖矿病毒事件。

# 2. 深入分析进程信息

有时候，top 的输出可能因为屏幕宽度限制还是无法显示全部命令。这时，ps 命令就派上用场了。

# 假设通过 top 查到进程的 PID 是 2094448
ps -p 2094448 -o cmd --no-headers

# 如果命令行太长，可以重定向到文件查看
ps -p 2094448 -o cmd --no-headers >> process_command.log

# 3. 寻找入侵途径

清除了病毒进程（kill -9 <PID>）只是第一步，如果不堵住漏洞，它很快就会卷土重来。我们需要弄清楚攻击者是如何进来的。

SSH 弱口令爆破是最常见的入侵方式之一。我们可以通过查看 SSH 的登录日志来寻找线索。

# 实时监控 SSH 登录日志
tail -f /var/log/auth.log

# 查看完整的日志文件，搜索失败的登录尝试
grep "Failed password" /var/log/auth.log

通过分析日志，我发现了大量的来自陌生 IP 的登录失败记录，紧接着是一个成功登录的记录。这证实了服务器的某个账户存在弱密码，被攻击者通过暴力破解成功登录。

# 4. 清除病毒与持久化机制

  1. 杀死进程：kill -9 <PID>
  2. 删除病毒文件：rm /tmp/networkSync
  3. 检查定时任务：攻击者通常会创建定时任务来实现病毒的持久化和“复活”。
    crontab -l
# 检查 /etc/crontab 以及 /etc/cron.* 目录下的所有文件
  4. 检查系统服务：更高级的病毒可能会将自己注册为系统服务。
    systemctl list-unit-files | grep enabled

# 安全加固措施

亡羊补牢，为时未晚。在清除了病毒之后，必须立即进行安全加固，防止再次被入侵。

# 1. 强化 SSH 安全

  • 禁用密码登录，强制使用密钥登录：这是最有效的防爆破手段。编辑 /etc/ssh/sshd_config，设置 PasswordAuthentication no。

  • 修改默认端口：将默认的 22 端口改为其他不常用的端口。

  • 禁用 root 用户直接登录：设置 PermitRootLogin no。

  • 配置登录白名单：使用 AllowUsers 指令，或者配置 hosts.allow 和 hosts.deny 来限制可以登录的 IP 地址。

    示例：使用 hosts.allow 和 hosts.deny

    a. 编辑 /etc/hosts.deny，禁止所有 IP 通过 SSH 登录：

    ```
sshd: ALL
```

    b. 编辑 /etc/hosts.allow，添加允许登录的白名单 IP：

    ```
# 只允许 192.168.1.3 这个 IP 登录
sshd: 192.168.1.3
```
系统会先检查 `hosts.allow`，如果匹配成功则直接放行；否则，再去检查 `hosts.deny`。

# 2. 定期审计与更新

  • 定期检查服务器上的用户和权限。
  • 及时更新系统和软件包，修复已知的安全漏洞。
  • 使用 fail2ban 等工具来自动阻止恶意爆破的 IP。

# 总结

这次挖矿病毒的排查是一次典型的服务器安全实战。它提醒我们，安全是一个持续的过程，而不是一次性的配置。从设置强密码、使用密钥登录，到配置防火墙规则和定期审计，每一个环节都不能掉以轻心。只有建立起纵深防御的体系，才能最大限度地保护我们的服务器免受侵害。
