# GitLab Runner配置与CI/CD实践

GitLab Runner是GitLab CI/CD的核心组件，负责执行CI/CD流水线中定义的任务。本文记录了使用Docker部署GitLab Runner并配置自动化构建的实践过程。

# GitLab Runner部署

# 使用Docker安装

推荐使用Docker方式部署，便于管理和维护：

# 创建配置目录
mkdir -p /data/gitlab-runner/config

# 启动GitLab Runner容器
docker run -itd --restart always --name gitlab-runner \
  -v /data/gitlab-runner/config:/etc/gitlab-runner \
  -v /var/run/docker.sock:/var/run/docker.sock \
  gitlab/gitlab-runner:v15.3.3

挂载说明：

  • /data/gitlab-runner/config: Runner配置文件目录
  • /var/run/docker.sock: Docker socket，允许Runner创建容器执行任务

# 注册Runner

容器启动后，需要在GitLab项目中注册Runner：

  1. 进入GitLab项目的Settings → CI/CD → Runners
  2. 复制注册token
  3. 执行注册命令：
docker exec -it gitlab-runner gitlab-runner register

按提示输入：

  • GitLab URL
  • Registration token
  • Runner描述
  • Runner tags
  • Executor类型（推荐docker）

# CI/CD配置示例

# Vue3项目自动化构建

以下是一个基于Node.js的Vue3项目自动化构建配置示例（.gitlab-ci.yml）：

image: node:16.18.0  # 使用Node.js镜像作为基础环境

stages:
  - build
  - deploy

variables:
  INSTALL_DIR: "/data/web"

cache:
  paths:
    - node_modules/

install-job:
  stage: build
  script:
    - echo "install..."
    - npm config set registry https://registry.npmmirror.com
    - npm install pnpm@8.7.6 -g
    - pnpm install
    - echo "build..."
    - pnpm run build
  cache:
    paths:
      - node_modules/
  artifacts:
    paths:
      - dist/

deploy-job:
  stage: deploy
  environment: production
  script:
    - echo "Deploying to ${INSTALL_DIR}..."
    - mkdir -p ${INSTALL_DIR}
    - cp -r ./dist/* ${INSTALL_DIR}

# 配置说明

stage阶段：

  • build: 安装依赖并构建项目
  • deploy: 将构建产物部署到指定目录

cache缓存：

  • 缓存node_modules目录，避免每次都重新安装依赖
  • 提升构建速度

artifacts制品：

  • 保存构建产物dist目录
  • 在后续stage中可以使用

variables变量：

  • 定义部署目标目录
  • 便于统一管理和修改

# 优化建议

  1. 使用特定的Node版本： 确保构建环境一致性
  2. 配置镜像源： 使用国内镜像源加速依赖下载
  3. 合理使用缓存： 缓存依赖包减少安装时间
  4. 限制Runner执行： 使用tags限定特定Runner执行任务

# 常见问题

# 磁盘空间被占满

GitLab Runner长期运行后，日志文件可能会占用大量磁盘空间。

问题定位：

# 查看Docker容器日志文件大小
du -sh /var/lib/docker/containers/*

解决方法：

# 删除指定容器的日志文件
# 容器ID可以通过 docker ps 查看
cd /var/lib/docker/containers/<container_id>/
> <container_id>-json.log

预防措施：

启动容器时限制日志大小：

docker run \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  ...

或在/etc/docker/daemon.json中全局配置：

{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}

# SSH非22端口克隆问题

当GitLab使用非标准SSH端口时，克隆地址格式需要调整。

错误示例：

git clone git@192.168.2.5:/museum/jiangle/jiangle_admin_web.git

正确格式：

# 使用 ssh:// 协议，并指定端口号
git clone ssh://git@192.168.2.5:8022/museum/jiangle/jiangle_admin_web.git

# Developer角色无法Push

Developer角色默认对protected分支没有push权限。

解决方法：

  1. 进入项目Settings → Repository → Protected Branches
  2. 找到需要修改的分支（如master）
  3. 在"Allowed to push"中添加Developer角色
  4. 或取消该分支的protected状态

注意： 生产环境建议保持master分支的保护状态，通过Merge Request合并代码。

# Git凭据管理

Windows系统中Git会缓存凭据，更换账号时需要清除缓存。

清除方法：

  1. 打开Windows凭据管理器
    • 控制面板 → 用户账户 → 凭据管理器
  2. 找到Git相关的凭据（如github.com或GitLab地址）
  3. 删除对应凭据
  4. 下次操作时会重新提示输入账号密码

命令行方式：

# 清除所有Git凭据
git credential-manager clear

# 或者配置不保存凭据
git config --global credential.helper ""

# 最佳实践

  1. 分支策略：

    • 保护主分支，通过MR合并代码
    • 使用feature分支开发新功能
    • 定期清理已合并的分支

  2. Runner管理：

    • 为不同项目配置专用Runner
    • 使用tags区分不同类型的任务
    • 定期清理旧的构建缓存

  3. 安全性：

    • 不在.gitlab-ci.yml中明文存储敏感信息
    • 使用GitLab CI/CD变量管理密钥
    • 配置变量时选择"Masked"和"Protected"

  4. 性能优化：

    • 合理使用缓存减少构建时间
    • 并行执行独立的任务
    • 使用镜像源加速依赖下载
